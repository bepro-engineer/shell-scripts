# Shell レビューワークフロー

## 目的

このファイルは、Shell スクリプトのレビュー時に行う作業手順を定義する。

## 前提

レビュー段階では、コード修正を行わない。

ユーザーが以下を依頼した段階では、確認と整理だけを行う。

- 整理
- 確認
- レビュー
- 設計
- 問題点抽出
- 修正方針
- task登録項目の整理

## 禁止事項

レビュー段階では、以下を行わない。

- ファイルを直接編集しない
- git diff が発生する変更を行わない
- コード整形を行わない
- リネームを行わない
- 並び替えを行わない
- 不要行削除を行わない
- 修正後コード全文を出力しない
- patch を出力しない
- diff を出力しない
- 直接編集しない

## レビュー時の確認手順

1. 対象ファイルを確認する

ユーザーが指定したファイルだけを対象にする。

2. 現在の実ファイル本文を確認する

過去の記憶や前回の回答だけで判断しない。

3. CLAUDE.md または rules 配下の根拠を確認する

指摘には必ず根拠を持たせる。

4. 問題点を抽出する

修正は行わず、問題点だけを整理する。

5. task 登録候補に分ける

修正単位が複数ある場合は、task に分けられる粒度で整理する。

## 出力内容

レビュー段階で出力してよい内容は以下に限定する。

- 問題点一覧
- 修正方針
- 修正対象関数
- task に分けられる修正項目一覧
- 想定変更点の要約

## レビュー観点

- shebang が `#!/bin/bash` になっているか
- 実行ファイル系 Shell Script で前処理に `startLog` があるか
- `startLog` が引数解析完了後、主処理開始前に1回だけ呼ばれているか
- Usage 表示用の関数が表示専用になっているか
- 引数不正時に当該関数内で `exitLog "${JOB_ER}"` により終了しているか
- 呼び出し元で戻り値判定方式を混在させていないか
- `-h` `--help` を実装している場合、Usage 表示後に `JOB_OK` で終了するか
- 引数なし・不正オプションで Usage 表示後に `JOB_ER` で終了するか
- 先頭行・最終行が正確に `--------------------------------------` になっているか
- Usage 表示例と1行単位で完全一致しているか
- 先頭が `-` の文字列を `printf` の format 文字列にしていないか

## 修正へ進む条件

実際のコード修正は、ユーザーが以下を明示した後にだけ行う。

- task 登録完了
- task start 完了
- feature ブランチ確認済み
