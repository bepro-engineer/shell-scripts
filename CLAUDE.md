# shell-scripts

## このリポジトリの目的

Shellスクリプト群の作成・整理・見直しを行う。

## Shellスクリプト共通ルール

- 新規作成するShellスクリプトは `#!/bin/bash` を使用する
- 新規作成時の標準は Bash とし、`sh` 互換は前提にしない
- 雛形は業務処理を含めず、骨組みだけを保持する
- 雛形にサンプル設定ファイル生成処理を書かない
- 雛形内の DEBUG 出力は `logOut "DEBUG"` を使用する
- `echo` によるログ出力は行わない
- trap は必ず `terminate` 関数を呼ぶ
- trap でシグナル `0` は捕捉しない
- trap は `HUP` `INT` `QUIT` `TERM` を使用する
- ロック取得前に trap を設定する
- 終了コードは `JOB_OK` `JOB_WR` `JOB_ER` で管理する
- 新規作成時は `templates/basicTemplate.sh` を雛形として使用する

### scope ルール

- `scope` は `var` `func` `pre` `main` `post` を使用する
- 変数定義ブロックでは `scope="var"` を設定する
- 関数定義ブロックでは `scope="func"` を設定する
- 前処理では `scope="pre"` を設定する
- 主処理では `scope="main"` を設定する
- 後処理では `scope="post"` を設定する

## 基本方針

このリポジトリ内の Shell Script 修正・追加・レビューでは、以下の規約を必ず優先すること。

- 修正は必ず関数単位で行う
- 既存コードの全面書き換えはしない
- まず現状整理、次に問題点抽出、その後に修正を行う
- 既存修正対象は都度指定されたファイルとする
- 新規作成時は用途に応じて `bin` `com` `etc` `log` `rep` `tmp` を使い分ける
- 指示されていないディレクトリ名・ファイル名・構造を勝手に新規作成しない
- 不明点は推測せず停止する
- 既存の関数名・変数名・責務を勝手に再設計しない
- 変更対象外の関数には触れない

## 命名規約

### ファイル名

- 新規作成する実行ファイル系 Shell Script 名はキャメルケースを使用する
- 既存の共通シェルスクリプト名は既存名称を維持する
- 実行ファイル系 Shell Script の拡張子は `.sh` を使用する
- 共通関数・共通処理などのリソース系 Shell Script の拡張子は `.shrc` を使用する

### 関数名

- 関数名は必ずキャメルケースを使用する
- 真偽値判定の関数名は `is...` を使用する
- 取得系の関数名は `get...` を使用する

### 変数名

- ローカル変数は `local` を付け、スネークケースを使用する
- グローバル変数は大文字スネークケースを使用する

### 定数名

- 定数は大文字スネークケースを使用する

## ディレクトリ規約

新規の Shell スクリプト作成時は、用途ごとに配置先を必ず分けること。
ファイルの役割に応じて、以下のディレクトリへ配置すること。

- 実行ファイルは `projects/scripts/bin`
- 共通シェルスクリプトは `projects/scripts/com`
- 設定ファイルは `projects/scripts/etc`
- ログ関連ファイルは `projects/scripts/log`
- 出力レポートなどは `projects/scripts/rep`
- 一時ファイルは `projects/scripts/tmp`

ファイルの役割と配置先が一致していることを必ず確認すること。
用途に合わないディレクトリへ勝手に配置しないこと。
新規ファイル作成時は、必ずこのディレクトリルールに従うこと。

共通スクリプトは以下を前提とする。

- ログ出力: `projects/scripts/com/logger.shrc`
- 共通処理: `projects/scripts/com/utils.shrc`

## 新規ファイル作成ルール

- Bash を標準とし、Bash 前提の実装で統一する
- 新規ファイル作成時は、必ず既存のディレクトリ規約・命名規約・コメント規約に従う
- 指示されていない新規ディレクトリは作成しない
- 新規作成する場合も、まず既存の `logger.shrc` と `utils.shrc` の利用可否を確認し、共通化できる処理を重複実装しない

## 実行ファイルルール

- 共通シェルスクリプト `.shrc` は単独実行を前提にしない
- 共通シェルスクリプト `.shrc` には実行権限を付与しない
- 実行権限は実行ファイル `.sh` にのみ付与する

## source ルール

- 共通関数を利用する Shell Script は、必要な `.shrc` を明示的に source して使用する
- ログ出力が必要な場合は `logger.shrc` を読み込む
- 共通処理を利用する場合は `utils.shrc` を読み込む
- 読み込み可能な前提を確認せずに関数を直接呼ばない

## ログ出力ルール

- ログ出力は必ず `logOut` を使用する
- `logInfo` や `logError` などの独自略称関数は使用しない
- 形式は必ず以下とする

```bash
logOut "INFO" "message"
logOut "WARN" "message"
logOut "ERROR" "message"
```

## コメントルール

- 新規ファイル作成時は、ファイル先頭に用途が分かるコメントを付ける
- 少なくとも、ファイル名、概要、役割、使用箇所が判別できる状態にする
- 既存ファイルにヘッダ形式がある場合は、その形式に合わせる
- すべての関数には必ずコメントを付ける
- 既存関数の修正時も、新規関数の追加時も、関数コメントは省略しない
- コメント未記載の関数を新規追加しない
- 関数コメントは日本語で統一する
- コメント形式は必ず以下のテンプレートを使う

```

# 関数名　　：xxx
# 概要　　　：○○○
# 説明　　　：

#
# 引数　　　：説明
# 戻り値　　：説明
# 使用箇所　：説明
# ------------------------------------------------------------------
```

## 実装ルール

- 不要な Bash 固有機能の濫用は避ける
- `echo -e` より `printf` を優先する
- グローバル変数への隠れ依存を避け、必要な値は引数で受け渡す
- 冗長な `return $?` は書かない
- 未使用関数は残さず、呼び出し有無を確認して整理する
- 責務が重複している関数は、共通処理へ寄せて整理する
- 文字列比較、数値比較、空判定は境界ケースを意識する
- `set -u` 環境で落ちないように引数不足を考慮する

## 修正ルール

- 修正対象ファイルだけを変更する
- 差分は最小にする
- コメントだけ直す場合も、実装との整合が取れていること
- 既存セクション名・関数名を勝手に変えない

## 検証ルール

修正後は必ず以下を行うこと。

1. 構文確認
   `bash -n 対象ファイル`

2. 差分確認
   `git diff -- 対象ファイル`

3. 必要最小限の動作確認
   対象関数を source して直接実行し、戻り値または出力を確認する

## Git ルール

- cc-tasks の開始から終了までの固定手順に従う
- 作業前に task を start する
- 作業は feature ブランチで行う
- 対象ファイルだけ add する
- main へ merge 後、push と branch 削除まで行う
- `.task/tasks.json` と `.task/config.json` は Git 管理対象とする
- `.task/` を `.gitignore` で除外しない
